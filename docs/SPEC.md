## What we are building

A **ZK Proof-of-Solvency** for a Stellar-issued asset / tokenized vault. A custodian/issuer proves ON-CHAIN that
its total liabilities (the sum of all customer balances) are `<= a PUBLIC threshold T` (solvency) **without
revealing any individual balance or the exact total**.

- **Off-chain** (Circom + Groth16): a circuit takes N private balances, builds a Merkle-Sum-Tree, Poseidon-commits
  to it (public `rootHash`), range-checks every balance to 64 bits (blocks negative/overflow field-wrap fakery),
  sums the tree, and proves `rootSum <= T`. Public inputs: `rootHash`, `T`.
- **On-chain** (Soroban): a verifier contract checks the Groth16 proof against `(rootHash, T)` and, on success,
  writes a **timestamped solvency attestation** `{root, threshold, ledger_timestamp, status: Solvent}` to storage
  and emits an event. On a failed proof the host pairing check fails, the tx reverts, no attestation is written.
  **That pass/fail dependence on a proof over data the chain never sees is the load-bearing ZK.**

Post-FTX proof-of-reserves framing. The **asset side** (issuer controls reserves `>= T`) is honestly
attested/mocked, not proven (the standard, disclosed PoR cut).

## Stack (risk-adjusted for solo + AI, new to ZK, 7 days)

- On-chain verifier = Stellar's OFFICIAL `soroban-examples/groth16_verifier` (~55 lines, `#![no_std]`, uses
  `env.crypto().bls12_381()` g1_mul/g1_add/pairing_check; generic over public-input count). Only edit to adopt it:
  bump `soroban-sdk` 25.1.0 → 26 to match this workspace; target `wasm32v1-none` already matches our loop.
- Off-chain pipeline cloned from `jamesbachini/CircomStellar` (`demo.sh` + `circom_to_soroban_hex` for the fiddly
  G1/G2/Fr serialization).
- Solvency circuit reference = **Summa** Merkle-Sum-Tree proof-of-solvency (trim to N=8, single asset).
- Rejected: RISC Zero (~35-min laptop proofs). Fallback: Noir/UltraHonk (Tier 3 below).

### The #1 risk: curve discipline
snarkjs defaults to BN254 and almost every Summa/circomlib reference uses BN254 Poseidon, but the Stellar verifier
is **BLS12-381**. Must compile `--prime bls12381`, run `powersoftau new bls12-381`, and use **BLS12-381 Poseidon
constants** (`jmagan/poseidon-bls12381-circom`), NOT circomlib's default BN254 Poseidon. Copy-pasting a BN254
circuit silently produces a wrong-field proof. **Isolated to Day 2 on a trivial circuit before the real one.**

## ZK green-gate (extends verify-soroban's green-lie guard)

A change is "verified" ONLY when, in the SAME run, BOTH are observed (neither alone is enough):
1. **POSITIVE** — a real honest Groth16 proof generated by OUR pipeline (not an un-regenerable fixture) verifies
   on-chain inside a passing Soroban test (the test must call `env.crypto().bls12_381().pairing_check` via the
   contract, not stub it), AND the `Attestation{root,threshold,timestamp,Solvent}` is read back from storage + the
   event asserted.
2. **NEGATIVE** — in the same suite, a tampered-proof case (flipped bytes → verify reverts, NO attestation written)
   AND an insolvent-input case (`sum > T` / wrong rootHash → rejected) both FAIL verification and are asserted.

Showing only the pass is a GREEN LIE. ALARM (treat as red): `0 passed`; a "passing" reject test that never actually
ran the verify (assert the specific revert, not bare `should_panic`); a proof that verifies after the circuit
changed but the vkey wasn't regenerated (stale-vkey false-green → pin vkey+proof+circuit hash together).

## Honest mocks (rules explicitly reward disclosure)

- Customer balances = a FIXED synthetic 8-entry CSV (no real customer/exchange data).
- Asset side (reserves `>= T`) is attested/placeholder, NOT cryptographically proven.
- Trusted setup is a SINGLE-PARTY (non-ceremonial) powers-of-tau + phase2 — sound enough to demo, disclosed as toy.
- N fixed at 8 leaves (bounded, not an unbounded FTX-grade tree).
- The contract is the official `groth16_verifier` ("demonstration only, not security audited") + a thin attestation
  layer. No production-solvency guarantees claimed.
- Single asset (no multi-asset collateral-haircut tiers).

## Fallback if ZK stalls (decided at the Day 2-3 gate)

- **Tier 1** (solvency circuit fights back, pipeline works): drop the Merkle-Sum-Tree, Poseidon-commit the balance
  VECTOR directly (`rootHash = Poseidon(balances)`), prove `sum(balances) <= T` with per-leaf range checks.
- **Tier 2** (BLS12-381 Poseidon constants are the blocker): keep the sum/range logic, commit with Pedersen/MiMC
  that has working BLS12-381 circom support, OR reduce N to 4.
- **Tier 3** (Circom/BLS12-381 stalls past Day 3): switch to Noir/UltraHonk (pins soroban-sdk 26 + wasm32v1-none =
  our exact stack, Rust-like, ready `tornado` Poseidon2 Merkle template). Measure `feeCharged` early; keep tiny.
- **Tier 4** (floor, still valid): verify a real BLS12-381 proof of a non-trivial private statement ("I know 8
  private values whose Poseidon-commitment is this public root and whose sum <= T") without the full tree; tree =
  future work.

In every tier: never claim it works without seeing a real verify PASS and a real REJECT.

## Why this also advances the main goal (#1 Contract Verification RFP)

The reusable artifact IS verification infrastructure: a deployed, tested Soroban contract that verifies an
off-chain computation on-chain + records a timestamped attestation is exactly the verify-and-attest primitive the
RFP is about (the Attestation storage + event layer is domain-agnostic). It upgrades `verify-soroban` into a ZK
green-gate, proves out the full off-chain→on-chain ZK toolchain once, and stress-tests the stellar-lab foundation
under a genuinely hard new domain. Even if the hackathon places nowhere, stellar-lab keeps a battle-tested on-chain
ZK verifier + a ZK-aware verify loop.
