## Â§0 NON-NEGOTIABLE RULES

1. **Green-lie guard (the #1 rule).** A change is "verified" ONLY when you observe BOTH real
   success signals in the SAME run:
   - Build: `stellar contract build` prints **`âś… Build Complete`** AND the wasm exists at
     `target/wasm32v1-none/release/<crate>.wasm`. A failed build prints `âťŚ error: exit status 101`.

   for the success TEXT above. If you must capture status, use `set -o pipefail` + `${PIPESTATUS[0]}`
   AND still confirm the text.

## Â§0.5 ZK green-gate (when the change involves a ZK proof / verifier)

On top of Â§0.1, a ZK change is "verified" ONLY when BOTH are observed in the SAME run:
- **POSITIVE:** a real proof generated by OUR pipeline (not an un-regenerable fixture) verifies on-chain inside a
  passing test that actually calls `env.crypto().bls12_381().pairing_check` via the contract (not stubbed); for
  attest flows, the `Attestation` is read back from storage + the event asserted.
- **NEGATIVE:** in the same suite, a tampered proof AND a wrong/insolvent public input are both REJECTED (verify
  returns `false` or reverts), asserted explicitly (assert the specific result, not bare `should_panic`).

Showing only the PASS is a GREEN LIE. ALARM (treat as red): `0 passed`; a reject test that never actually ran the
verify; a proof that still verifies after the circuit changed but the vkey wasn't regenerated (stale-vkey
false-green â€” pin circuit + vkey + proof together as ONE artifact set). The official `groth16_verifier` returns
`Ok(false)` for a bad proof (it does NOT revert), so the base reject assertion is `assert_eq!(res, false)`.
